Chapter: The Problem
Chapter Slide 3 THE PROBLEM WITH MANUAL DEPLOYS
Let this sit for 2–3 seconds. The dark background is a visual cue that something important is coming.
Content Slide 4 MANUAL DEPLOYS BREAK PRODUCTION ~3 min
🎤 Open with a question
Before I go through these cards, let me ask: when someone on your team needs to add a column to a production table today, what does that actually look like? Who runs the SQL? How do you know it happened?

Pause. Let them answer. Then walk each card:

🎤 Card 1 — No Audit Trail
If something breaks at 2am — and it will — you can't trace it. "Who changed the ORDERS table?" The answer is a Slack thread from three weeks ago, if you're lucky. Git solves this completely. Every change is tied to a commit, an author, and a timestamp.
🎤 Card 2 — Environment Drift
This one kills teams quietly. Dev, staging, and prod start identical. Then six months later someone runs an ALTER TABLE in prod directly. Someone else fixes a bug in dev but forgets to apply it to staging. The environments silently diverge. You stop trusting your tests because dev doesn't match prod.
🎤 Card 3 — Silent Pipeline Breaks
This is the most dangerous one for a data engineering team. Your Airflow DAGs reference specific column names. Someone renames a column in Snowflake without telling the DAG owner. The pipeline fails silently at 3am and nobody knows until the business reports look wrong. With CI/CD, the schema change and the DAG fix ship together in one PR.
❓ They might say
"We have change management processes for this." — Acknowledge it, then point out that process-based governance breaks under pressure. CI/CD makes the right path the easy path. You don't rely on people to remember to follow the process.
Chapter: The Approach
Chapter Slide 5 THE APPROACH
A brief pivot moment. Say something like: "So what's the answer?"
Content Slide 6 ONE REPO. ONE SOURCE OF TRUTH. ~3 min
🎤 Say
The solution is simple in concept. Four principles. Number one: everything is code. If you didn't commit it, it doesn't exist. No more clicking in the Snowflake UI to create tables. Number two: changes flow through environments. Dev to staging to prod, in that order, never backwards. Number three: CI/CD automates deployment. The pipeline validates your SQL, tests your DAG syntax, and deploys — all triggered by a merge, no human in the loop. Number four: secrets stay out of Git. Credentials are injected at runtime from a secrets manager. Never hardcoded, never committed.
✦ Best practice to emphasize
We're using a monorepo — one repo for both Snowflake schemas and Airflow DAGs. This is intentional. These components are tightly coupled: a schema change and a DAG change often need to ship together. One PR, one review, one deploy. If you separate them, you create coordination overhead without the benefit.
❓ They might ask — "Is a monorepo the right pattern for us?"
For most teams of 2–15 engineers: yes. You can always split later as you scale. Starting split creates coordination overhead you don't need yet. The test is: do your Snowflake schemas and DAGs change together? If yes, keep them together.
Chapter: Core Capabilities
Chapter Slide 7 CORE CAPABILITIES
"Let me show you the three building blocks that make this work."
Content Slide 8 SNOWFLAKE READS FROM GITHUB NATIVELY ~2 min
🎤 Say
Most people don't know this feature exists. Snowflake has a first-party Git integration — you run one SQL command, link your GitHub repo, and Snowflake can pull files directly from it. No middleware. No custom tooling. It works with GitHub, GitLab, and Bitbucket. Once it's linked, you can execute SQL scripts from within Snowflake itself, referencing the file by its path in the repo. We'll see this live in a moment.
🎤 Walk the right column
What can you do with it? Execute migration scripts, deploy stored procedures and UDFs, sync stage files. And you can trigger a fetch from a Snowflake Task — so your scheduled jobs always run the latest version of the code from Git.
✦ Key distinction to make
Snowflake Git Repository and GitHub Actions CI/CD are complementary, not competing. Git Repository is great for ad-hoc execution and Tasks. CI/CD is the right tool for structured, environment-promoted schema migrations. You'll likely use both.
Content Slide 9 SCHEMA CHANGES ARE MIGRATIONS, NOT EDITS ~2 min
🎤 Say
The tool that handles the Snowflake side is called schemachange — it's open source, built by Snowflake Labs. The pattern is called migration-based versioning, and it has one golden rule: you never edit a file that's been deployed. You only add new ones.
🎤 Point to the code block on the slide
Look at the file names. V1.0.0, V1.1.0, V1.2.0. That version prefix isn't optional — schemachange runs files in this exact order and tracks which ones have already been applied in a table called CHANGE_HISTORY inside Snowflake. If you try to run them out of order, it rejects it. If a file has already run, it skips it. The history is the source of truth for what's been deployed.
🎤 Point to the right-side annotations
Three things to remember: version your files with a consistent format, never edit deployed migrations, and connect schemachange to GitHub Actions so it runs automatically on merge. That's it.
❓ "What if a migration fails halfway?"
schemachange records the failure in CHANGE_HISTORY. The next run will retry it. You fix the SQL and redeploy. This is why each migration should use IF NOT EXISTS or IF EXISTS guards where possible — it makes them safe to re-run.
Content Slide 10 ONE REPO GOVERNS BOTH SIDES OF YOUR PIPELINE ~2 min
🎤 Say
Here's the actual folder structure of our demo repo. Four folders, four responsibilities. /snowflake/ for SQL migrations — schemachange owns this. /dags/ for Airflow DAG files — CI/CD syncs these to your Airflow environment on deploy. /tests/ for validation scripts — these run on every PR to catch issues before they reach main. And /.github/workflows/ — this is where the automation lives. We'll look at that file in a moment.
✦ Emphasize this
Nobody on your team ever touches Snowflake directly to make schema changes. If you need a new table, you create a file in /snowflake/, commit it, and let the pipeline handle the rest. That's the cultural shift. The tooling enforces it.
Chapter: End-to-End Pipeline
Chapter Slide 11 END-TO-END COMMIT TO PRODUCTION
"Let me show you the full flow from a single commit to a live Snowflake change."
Content Slide 12 FROM COMMIT TO PRODUCTION IN 5 STEPS ~2 min
🎤 Walk each chevron left to right
Step 1: a developer creates a migration file and opens a Pull Request. Step 2: GitHub Actions automatically validates the SQL naming convention and runs tests — no human needed. Step 3: a teammate reviews the PR and it gets merged to main. Step 4: the deploy job fires automatically — schemachange connects to Snowflake and runs any new migrations. Step 5: the same pipeline syncs DAG files to Airflow.
🎤 Point to the bottom banner
This isn't just for schema changes. New stored procedure? Same flow. Updated DAG? Same flow. New view? Same flow. One process handles everything, consistently, every time.
✦ Best practice to say out loud
The key architectural decision here is: the deploy only happens on merge to main. You never run schemachange locally against production. The pipeline is the only thing that touches prod. This eliminates the "I ran something manually and forgot to commit it" problem entirely.
Chapter: Comparison
Chapter Slide 13 CI/CD VS. MANUAL
"Let's be concrete about what the difference actually looks like day to day."
Content Slide 14 CI/CD VS. MANUAL: EVERY STEP MATTERS ~2 min
🎤 Walk the table
Look at the left column. Version history: with manual deploys, your audit trail is Slack messages and tribal knowledge. With CI/CD, it's a Git commit log with author, timestamp, and a diff showing exactly what changed. Environment promotion: manual means copy-pasting SQL between environments and hoping you got it all. CI/CD means a PR merge automatically promotes the change through dev, staging, and prod in the right order. Secrets: hardcoded credentials in scripts versus encrypted secrets injected at runtime that never touch your codebase. The right column is just strictly better across every dimension.
💡 Tip
Let them read the table. Don't rush. The contrast is stark and it does the work for you. If someone asks "can I see the rollback row?" — say: with CI/CD, rollback is just reverting the PR and re-running the pipeline. That revert is also in Git history.
Chapter: Architecture
Chapter Slide 15 THE ARCHITECTURE
"Here's what this looks like as a system diagram."
Content Slide 16 GITHUB IS THE CONTROL PLANE ~2 min
🎤 Top to bottom
GitHub is the control plane. It's where all your code lives and where all automation originates. GitHub Actions is the engine — it runs on every push and handles both validation and deployment. Then two deployment targets: Snowflake on the left, Airflow on the right. Snowflake gets your schema migrations and SQL objects. Airflow gets your DAG files. They both receive changes from the same pipeline, triggered by the same PR merge.
🎤 Point to the left box
Notice that Snowflake also has its own Git Repository object — it's separately linked to GitHub. This gives Snowflake the ability to pull and execute files directly, independent of the CI/CD pipeline. Two integration surfaces, different use cases.
✦ Security point to raise
The CI/CD user that connects Snowflake to GitHub Actions is a dedicated service account with minimal permissions — only the CICD_DEMO database, only the AIRFLOW_WH warehouse. It uses key-pair authentication, which bypasses MFA requirements and doesn't expire like passwords. That's the production-grade approach for any programmatic Snowflake connection.

⌨️ Step 1 — Terminal Terminal
cat > snowflake/V2.0.0__add_email_to_customers.sql << 'EOF'
-- Migration: V2.0.0
ALTER TABLE RAW.CUSTOMERS ADD COLUMN IF NOT EXISTS email VARCHAR(255);
EOF
🎤 While typing
I'm creating a new migration file. Notice I'm not running any SQL. I'm just writing a file with a version number. The pipeline handles execution.
⌨️ Step 2 — Terminal Terminal
git checkout -b feature/add-email-column
git add snowflake/V2.0.0__add_email_to_customers.sql
git commit -m "Add email column to customers table"
git push origin feature/add-email-column
gh pr create --title "Add email column" --body "Adds email to CUSTOMERS"
🎤 While pushing
I've pushed to a feature branch. Nothing is deployed yet. Now I'll open a Pull Request — show the GitHub PR page. Notice CI is already running. That's the Validate job checking my file names. It's automatic.
🌐 Step 3 — Browser Browser
Switch to GitHub in the browser. Show the PR with the green check. Then merge it. 
gh pr merge --merge   # or click Merge on GitHub
🎤 After merging
The merge to main just triggered the Deploy job. Let's watch it.
🌐 Step 4 — Browser Browser
Switch to the Actions tab on GitHub: github.com/ppham0203/snowflake-cicd-demo/actions
Watch the "Deploy to Snowflake" job complete. Point out each step as it runs.
❄️ Step 5 — Snowsight Snowsight
Open Snowsight and run: 
DESCRIBE TABLE CICD_DEMO.RAW.CUSTOMERS;
Then show the audit trail: 
SELECT VERSION, SCRIPT, STATUS, INSTALLED_ON
FROM CICD_DEMO.RAW.CHANGE_HISTORY
ORDER BY INSTALLED_ON;
🎤 At the end of the demo
The EMAIL column is live. No one logged into Snowflake and ran ALTER TABLE. No one copy-pasted SQL between environments. It took less than 60 seconds from commit to production, and we have a complete audit trail in the CHANGE_HISTORY table that shows exactly what ran, when, and with what version.
❄️ Bonus — Snowsight Snowsight
-- In Snowsight:
ALTER GIT REPOSITORY CICD_DEMO.RAW.CICD_REPO FETCH;
LS @CICD_DEMO.RAW.CICD_REPO/branches/main/;
🎤 For the bonus
And here's the native Snowflake side. Snowflake can see every file in the repo directly. You can list them, execute them, or reference them from a Snowflake Task. This is how you'd deploy stored procedures or run ad-hoc scripts without leaving Snowflake.

Likely Questions
❓ "How do we handle rollbacks?"
schemachange is forward-only by design. To rollback, you add a new migration that undoes the change — V2.1.0__remove_email_column.sql with ALTER TABLE ... DROP COLUMN. This is intentional: the rollback itself becomes part of the audit trail.
❓ "How do we manage dev / staging / prod?"
Three options: (1) branch-per-environment — develop branch deploys to dev database, main deploys to prod; (2) parameterized migrations with Jinja variables; (3) separate Snowflake accounts per environment. Most teams start with option 1 — it's the simplest.
❓ "What about dbt?"
dbt and schemachange are complementary. schemachange handles DDL (CREATE TABLE, ALTER TABLE, GRANT). dbt handles transformations (SELECT-based models). Many teams use schemachange for the raw layer and dbt for the transform layer. They can live in the same monorepo.
❓ "Isn't this overkill for a small team?"
The setup takes half a day. The payoff is permanent. You'll never spend a Friday afternoon trying to figure out why prod doesn't match dev again.
❓ "What if our DBA doesn't know Git?"
The learning curve is one command: git add, git commit, git push. The PR review workflow is the same as a code review. Most modern DBA tooling has Git built in. The bigger shift is cultural — agreeing that the UI is read-only.
